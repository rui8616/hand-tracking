## 1. 产品定位

**叙事**：**屏幕时间健康助手**（Screen Time Health Assistant）

**一句话**：孩子用屏幕（网课、写作业、看动画）时，浏览器在后台用摄像头监测用眼姿势 — 脸距、低头、揉眼、不眨眼、光线不足 — 温和提醒，并给家长生成可量化的日报。

**核心叙事公式**：
> 「孩子用屏幕是**不可避免的**。既然必须用，就要用得健康。」

**这是什么**：
- 不打扰孩子学习、能陪伴使用的「数字护眼卫士」
- 家长可以「看到自己做了点什么」的安抚型工具
- **100% 本地处理摄像头数据**的隐私优先工具
- 一个**焦虑安抚 + 行为可视化**产品

**这不是什么**（明确划清）：
- ❌ 不是医疗设备、不诊断、不治疗、**不承诺降度数**
- ❌ 不是近视防控（这是个伪命题 — 户外光照才解决近视）
- ❌ 不是远眺训练（让孩子站远点看屏幕 ≠ 不看屏幕）
- ❌ 不是家长监控 App（不录像、不能远程查看孩子在做什么）
- ❌ 不是反屏幕工具（不强制孩子停用）

**为什么家长会付费**：
- 「孩子写作业时凑得太近」是几乎每个家长亲眼看到过的痛点
- 单靠口头催促 → 效果差 → 焦虑
- 量化数据 + 温和提醒 + 报告 = **「我尽力管理好了」的心理交代**
- 这不是「疗效」产品，是「焦虑安抚」产品 — 承认这个定位反而更诚实，营销也更好做

## 2. 用户画像与场景

**主要用户**：6-15 岁中小学生家长

**典型场景**：
- 孩子在家用 iPad / 笔记本写作业、上网课、看动画片
- 笔记本前摄或 USB 摄像头朝向孩子
- App / Web 在后台运行，孩子可以最小化或切换到学习 App
- 监测全程在本地，无网络也能用

**反场景（明确不做）**：

- 户外活动：超出 App 能力范围
- 集体场景（教室）：v3 再考虑

## 3. 5 个监测项的算法

技术基础：**MediaPipe Face Landmarker**（478 点脸部 + head pose matrix）+ 已有 Hand Landmarker。

### 3.1 脸到屏幕距离

**算法**：用瞳距反推距离。
- 取两眼瞳孔 landmark（468 左 / 473 右）的像素距离 `pxIPD`
- 已知儿童平均瞳距（IPD）：6-12 岁 ~55 mm、12-15 岁 ~60 mm
- 摄像头焦距 `f` 通过校准估算（首次启动让孩子站在 50 cm 位置点确认，反算 f）
- 距离公式：`distance_cm = (f_px * realIPD_mm) / pxIPD / 10`

**阈值**（动态可调）：
- > 40 cm：绿（正常）
- 30-40 cm：黄（提醒）
- < 30 cm：红（强提醒）

**平滑**：3 秒中位数滑窗，避免瞬时抖动触发误报。

### 3.2 头部姿态（低头 / 偏头）

**算法**：MediaPipe Face Landmarker 输出 4×4 transformation matrix → 提取 Euler angles（pitch / yaw / roll）。

**阈值**：
- pitch（低头）> 25° 持续 5 秒 → 提醒
- yaw（偏头）> 30° 持续 10 秒 → 提醒
- roll（歪头）> 20° 持续 10 秒 → 提醒

### 3.3 眨眼频率 / EAR（Eye Aspect Ratio）

**算法**：
- 左眼上下：landmark 159 ↔ 145；左右：33 ↔ 133
- EAR = ((|159-145| + |158-153|) / 2) / |33-133|
- 正常 EAR ≈ 0.25-0.3；眨眼瞬间 < 0.2
- 一次完整眨眼：连续 2-3 帧 EAR < 0.2 然后回升

**阈值**：
- 正常成人每分钟 12-20 次眨眼
- 看屏幕时降到 5-7 次（公认现象）
- **每分钟 < 5 次持续 3 分钟 → 提醒「眨眨眼」**
- **EAR 30 秒滑窗均值持续下降 → 干眼信号**

### 3.4 揉眼检测

**算法**：手部 21 点 + 脸部 478 点联合判断。
- 取食指尖（hand-8）和中指尖（hand-12）的像素位置
- 取眼睛中心（face-468、473）
- 当任一指尖与眼睛距离 < 该眼宽度的 60%，持续 > 0.8 秒 → 计为一次揉眼

**记录**：每天累计次数（不弹窗，只统计），> 5 次/天 → 报告标红「可能眼疲劳」。

### 3.5 环境光照监测

**算法**：从视频帧的非脸部区域（背景）取平均 luminance。
- L = 0.2126·R + 0.7152·G + 0.0722·B
- 背景区域：脸部 bbox 外的左上 + 右上 + 左下 + 右下角各 10% 区域
- 取平均后映射到 0-255

**阈值**：
- L < 80：提醒「光线偏暗，请开灯」
- 但要避开「白天孩子坐在窗前逆光」的误报 → 同时检测脸部亮度
- 1 分钟滑窗，避免频繁提醒

**注意**：浏览器拿不到摄像头真实曝光值（EV / ISO），自动曝光会补偿，这是不准的源头。要做用户引导：「请坐在你日常学习的位置，让 App 学习正常光照」（校准）。

## 4. 技术架构

### 4.1 模块拆分

```
src/posture-monitor/
├── face-tracker.ts        # MediaPipe Face Landmarker 封装
├── detectors/
│   ├── distance.ts        # 3.1 脸距
│   ├── pose.ts            # 3.2 头部姿态
│   ├── blink.ts           # 3.3 眨眼
│   ├── rubbing.ts         # 3.4 揉眼（需 hand + face）
│   └── lighting.ts        # 3.5 光照
├── state.ts               # 状态聚合 + 事件流
├── storage.ts             # IndexedDB 持久化
├── report.ts              # 日报 / 周报数据生成
├── ui/
│   ├── monitor.html       # 主监测页
│   ├── monitor.ts         # 实时反馈 UI
│   ├── calibrate.ts       # 首次启动校准流程
│   └── report.ts          # 报告页 UI
└── main.ts
```

### 4.2 复用 vs 新建

| 模块 | 复用 | 新建 |
|---|---|---|
| `hand-tracker.ts` | ✅ 直接用，揉眼检测需要 | |
| `hand-skeleton.ts` | ❌ 不需要 3D 渲染 | |
| Three.js | ❌ 不需要 | |
| MediaPipe Face Landmarker | | ✅ 加载 `face_landmarker.task` 模型 |
| IndexedDB 封装 | | ✅ 用 `idb` 库（轻量） |
| 图表（report） | | ✅ Chart.js 或 D3-mini |

### 4.3 性能预算

- 每帧需要跑 Face + Hand 两个模型 → 总耗时 ≤ 16ms 才能 60fps
- 实际：Face Landmarker GPU ~ 8ms，Hand ~ 6ms，余量足够
- 后台运行时降到 5-10 fps 即可（监测不需要 60fps）
- CPU 占用目标：< 15%（中端笔记本）

### 4.4 入口设计

两种打开方式：
1. **网页 PWA**：用户加书签 / 装到桌面，孩子学习前打开
2. **浮窗模式**（v1.5）：always-on-top 小窗口，不打扰学习

第一版只做网页方式，PWA installable，能离线运行。

## 5. 数据模型

```ts
type EventType = "distance" | "pose" | "blink" | "rub" | "light";
type Severity = "info" | "warning" | "alert";

interface PostureEvent {
  ts: number;            // unix ms
  type: EventType;
  severity: Severity;
  value: number;         // e.g. distance cm, pitch degrees, EAR
  durationMs: number;    // duration the violation lasted
}

interface SessionStats {
  sessionId: string;
  startTs: number;
  endTs: number;
  totalSeconds: number;
  avgDistanceCm: number;
  minDistanceCm: number;
  blinkPerMinute: number;
  rubCount: number;
  events: PostureEvent[];
}

interface DailyStats {
  date: string;          // YYYY-MM-DD
  sessions: SessionStats[];
  totalUsageMinutes: number;
  warningCount: Record<EventType, number>;
  pomodoroCompletions: number;
}

interface ChildProfile {
  id: string;
  name: string;
  ageYears: number;
  realIPDmm: number;     // calibrated or estimated from age
  cameraFocalPx: number; // from calibration
  createdAt: number;
}
```

存储：**IndexedDB**（不用 localStorage，容量小且同步）。

## 7. 隐私设计（卖点）

| 数据 | 处理位置 | 持久化 |
|---|---|---|
| 摄像头原始帧 | 内存，每帧用完即丢 | ❌ |
| MediaPipe 推理结果（landmarks） | 内存 | ❌（仅用于实时计算） |
| 监测事件 / 统计 | 浏览器 IndexedDB | ✅ 本地 |
| 用户档案 / 设置 | 浏览器 IndexedDB | ✅ 本地 |

> 🔒 摄像头数据 100% 在你的设备处理，不上传任何画面或脸部信息。

## 10. 定价模型（小而美定位）

按 **B 类型（小品牌精品）** 目标，定价压低、续费率优先、免费层真好用：

| 等级 | 价格 | 功能 |
|---|---|---|
| 免费 | 0 元 | 实时监测 + 当日报告 + 1 孩子档案 + 20-20-20 番茄钟 |
| 高级 | 9.9 元/月 或 88 元/年 | 历史报告（30 天+）+ PDF 导出 + 多孩档案（最多 2 个）+ 自定义阈值 |
| 家庭 | 19.9 元/月 或 168 元/年 | 上述全部 + 最多 4 孩 + 自定义提醒规则 + 优先客服 |

**为什么这个价格**：
- 中国家长教育类 App 心理价位 28-168 元/年区间（参见 product-research § 3.3）
- **88 元/年 ≈ 一杯咖啡的钱**，是「无脑订阅」的甜蜜点
- 让免费层真的够用 → 口碑传播 → 一部分人为方便/支持升级
- 不追求 ARPU 高，追求**续费率 > 60%**

**冷启动**：内测期 6 个月所有功能免费，邀请前 100 个家庭，留下反馈和留存数据。

**单位经济测算（B 类型现实目标）**：
| 阶段 | 目标 | 月流水 | 备注 |
|---|---|---|---|
| 6 个月内 | 30 付费用户（免费 → 付费转化测试） | ~300 元 | PMF 验证 |
| 12 个月 | 300 付费用户 | ~3000 元 | 第一个续费周期跑通 |
| 24 个月 | 1500 付费用户 | ~15000 元 | 1 人独立开发可全职 |
| 36 个月 | 5000 付费用户 | ~50000 元 | 1-2 人小团队稳定运营 |

**5000 付费用户 = 这个产品的"小而美"上限假设**。超过这个量级要么进化产品形态，要么承认市场到顶。

## 11. 产品演进路径（独立产品视角）

**核心约束**：守住「小而美」定位，不被复杂功能吞噬。

```

              3 周开发，6 个月内测调优 + 拿到付费验证
                ↓ 触发条件：500 付费用户 + D30 留存 > 15%

              不需大屏；同摄像头扩展，做眼球运动训练
                ↓ 触发条件：用户主动要求 + iOS Safari 摄像头权限改善
v2          — Native App（iOS / Android）+ 旧手机当摄像头模式
              扩到「孩子用平板写作业 + 旁边手机做摄像头」
                ↓ 触发条件：v2 流水稳定 6 个月
v2.5        — 多产品矩阵：成人版（程序员/办公族颈椎和眼疲劳）
              复用算法，不同场景包装
```

### 永远不做（明确划红线）

| 不做 | 原因 |
|---|---|
| 远眺训练大屏产品 | § 6 已证伪，伪需求 |
| 学校 / 教育局 B 端 SaaS | 不是这个团队的活，回款慢、客单大、销售重 |
| 「治近视」宣传 | 中国《广告法》合规风险，会被处罚 |
| 拿融资扩张 | 破坏小而美定位、上桌后被股东推着冲增长 |
| 抖音 / 小红书投放 | CAC > LTV，必死 |
| 学习内容（学科、英语等） | 双减红线 + 偏离产品定位 |
| 实时通话 / 远程监督 | 隐私焦虑反噬产品信任 |

| 教育博主 / 妈妈博主 | 季度合作 | 寄码、做评测、不投硬广 | ⭐ |

### 内容主题（直接可写的角度）

1. 「我家娃每天看屏幕 5 小时，我用一周的真实数据吓了自己一跳」
2. 「孩子写作业总凑得太近，我装了个 App 之后发现…」
3. 「市面上所有'护眼 App'都没用，因为它们假装不知道孩子还在看屏幕」（自我批判式）
4. 「100% 本地处理：孩子的脸的数据不会跑到任何服务器，这是我们的承诺」
5. 「不是治近视，是让屏幕时间变得健康一点 — 我们的诚实定位」

### 转化漏斗

```
看到内容 → 试用 PWA（零门槛、无需下载）
   ↓ 7 天后第一份周报
报告够好看 / 数据够吓人 → 升级付费（88 元/年）
   ↓ 第一个续费周期
分享给其他家长（核心传播节点）
```

### 关键 KPI（不是 DAU，是续费）

| 指标 | 优先级 | 目标 |
|---|---|---|
| 试用 → 付费转化 | ⭐⭐⭐ | > 5% |
| **续费率（年付）** | ⭐⭐⭐⭐ | > 60% |
| 自传播系数 K | ⭐⭐ | > 0.3（每用户带 0.3 个新用户）|
| 内容篇均阅读 | ⭐ | 5000+（小红书）/ 2000+（公众号） |
| NPS（净推荐值） | ⭐⭐ | > 40 |

**核心信念**：在 B 类型路线下，**1 个续费用户的价值 > 10 个流失用户的注册**。所以做好产品 + 做好内容 > 做好投放。

### 不要做（再次强调）

- ❌ 抖音 / 小红书信息流投放
- ❌ 拉新返现 / 邀请赠送会员
- ❌ B 端学校销售（这是另一个生意）
- ❌ 「治近视」宣传（合规风险）
- ❌ 上架 Apple Store / Google Play 同步推广（PWA 先跑通）

## 13. 几个开放问题

1. **校准能不能省掉？** —— 用儿童标准 IPD 表估算可以省，但精度差，误报率会高。先做"可选校准"。
2. **是否在浏览器跑？** —— 浏览器 PWA 是冷启动最快的方案。如果要做手机端，建议第一版仍是网页（响应式），用户用平板 / 旧手机当摄像头都行。Native App 是 v1.5 + 的事。
3. **要不要做"班级/集体"模式？** —— v1 不做。但保留架构空间，未来可以一个摄像头看多张脸。
4. **离线 PWA vs 联网订阅** —— 完全离线可用是卖点，订阅作为可选（数据云端备份 + 多设备同步）。
5. **产品名字** —— 还没敲定。候选：「看远点」「眼眼」「ScreenLook」「护眼小卫士」「屏幕同学」。建议先用工程代号「PostureWatch」开发，正式发布前定品牌名。
6. **付费用什么收？** —— 微信支付 / 支付宝直连最方便。第一版可以用"激活码"模式：手动发码，避免上来就接复杂的订阅引擎。

## 15. 决策红线（写给未来的自己）

**永远不要被沉没成本绑架。**

3 周开发完成 + 6 个月内测后看以下硬指标：

| 指标 | 红线 | 不达标的行动 |
|---|---|---|
| D7 留存 | < 30% | 砍掉、重新设计或放弃方向 |
| 付费转化率 | < 3% | 砍掉、重新定价或放弃方向 |
| 续费率（年付到期） | < 50% | 砍掉、重做核心价值 |
| 月被动新增（无投放） | < 30 | 内容策略失败，重做营销 |

**任一红线撞墙就停下来重新评估**，不要硬着头皮做完整产品。
